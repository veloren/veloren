# Nova-Forge Parallel Conversion Framework v1

**Policy status:** Active  
**Adopted:** April 2026

---

## Purpose

This document defines the dual-track strategy for converting Nova-Forge from its current playable base (inherited engine foundation) into a fully original Nova-Forge game identity, without destroying what already works.

The guiding principle is: **preserve, extend, then absorb** — never break the working game to speed up the new one.

---

## Two Runtime Lanes

### Track A — Stable Lane (Playable Baseline)

The current game as it exists today.

**Purpose:**
- Remains the default, shippable, playable experience at all times
- Serves as the design reference and usability benchmark for every new Nova-Forge system
- Keeps all inherited systems alive until their replacements are proven
- Provides movement feel, HUD conventions, combat baseline, interaction rhythm, and multiplayer runtime behaviour as a living comparison point

**Rules:**
- Track A is never broken intentionally
- Track A receives bug fixes and incremental quality-of-life improvements
- No Track B system replaces a Track A system until the replacement is complete and at least as usable
- Track A is the fallback if an Experimental world generation session is abandoned

---

### Track B — Experimental Lane (Nova-Forge Conversion Path)

The new game loop, exposed behind an **Experimental** world generation option.

**Purpose:**
- New world generation pipeline (Nova-Forge terrain, biomes, site archetypes)
- Nova-Forge economy: mining, salvage, merchant loops, blueprint research
- Home-base / hangar progression
- Deeper system replacement as each component matures
- The eventual default once it reaches Track A parity

**Rules:**
- Accessible only via the Experimental toggle in the world creation screen
- Clearly labelled in-UI as "under construction" / pre-release
- Does not touch the Track A world generation path
- Is not a dumping ground for unfinished ideas — it specifically means "the Nova-Forge replacement lane"

---

## Shared Systems (Reuse First Policy)

Before writing any new system in Track B, evaluate what already exists in Track A using this priority order:

| Fit | Action |
|-----|--------|
| Already fits Nova-Forge needs | **Reuse as-is** |
| Partly fits | **Wrap or extend** — add a thin layer without modifying the core |
| Does not fit | **Build alongside** — new implementation in Track B, old one survives |
| Old one is obsolete once new is proven | **Schedule absorption** — retire the old one in a named milestone |

This prevents two unrelated games from living in the same repository.

---

## Replace-Later Systems

Systems known to need eventual replacement but not yet ready:

| System | Track A state | Track B target | Replace when |
|--------|--------------|----------------|--------------|
| World generation | Inherited terrain pipeline | Nova-Forge worldgen (Experimental) | Track B worldgen feature-complete |
| Economy | None / barter only | Mining → Salvage → Blueprint loop | First playable Nova-Forge economy loop |
| Skill trees | Inherited ability system | Nova-Forge talent / research trees | New tree has at least 2 complete paths |
| Site generation | Inherited town/dungeon sites | Nova-Forge city/station archetypes | Nova-Forge sites are generated by default |
| Combat identity | Fantasy melee baseline | Nova-Forge-specific combat (TBD) | After Nova-Forge setting is locked |

---

## Absorption Criteria

A Track B system is ready to absorb its Track A counterpart when it meets **all** of the following:

1. **Feature parity** — covers every user-facing capability the Track A version provides
2. **Stability** — no crash-class bugs in 2+ consecutive release builds
3. **Usability parity** — feels at least as responsive and clear as the Track A version (judged by comparison, not gut feel)
4. **Test coverage** — has at least the same automated test surface as the replaced system
5. **Replacement ledger entry** — the specific Track A file/module being retired is listed and reviewed

Absorption is recorded in `CHANGELOG.md` as a named milestone entry.

---

## Experimental World Generation Tab

The Experimental lane is surfaced to players via a dedicated toggle in the world creation UI.

**Behaviour:**
- Default world creation uses the Track A (stable) pipeline
- An **Experimental** section or tab exposes Nova-Forge worldgen options
- The Experimental path displays a clear "under construction" notice
- Experimental worlds are not guaranteed save-compatible across development builds
- The toggle does not affect singleplayer worlds already created on the stable path

**Implementation note:** The world generation path selection should be stored in `WorldMeta` so that on load the correct pipeline is selected automatically without user intervention.

---

## Merge Triggers

The Stable and Experimental lanes converge — they do not split forever. The merge point is reached when:

- Track B worldgen is the new default (Experimental toggle removed, old pipeline archived)
- All Replace-Later systems in the table above have been absorbed
- The Nova-Forge content identity (lore, setting, progression) is coherent from session start to end
- Proper credits for the inherited foundation are permanently embedded in the in-game credits screen and `README.md`

---

## Failure Modes to Avoid

| Risk | Mitigation |
|------|-----------|
| Permanent split-brain: two games maintained forever | Define and enforce absorption criteria; keep the replacement ledger current |
| Copying instead of integrating: Track B duplicates Track A code | Apply the Reuse First Policy on every new Track B feature |
| Experimental becoming a dumping ground | Experimental means the Nova-Forge replacement lane only — not general WIP features |
| Breaking Track A while building Track B | Track A CI checks run on every commit regardless of which lane the change targets |

## Credits Commitment

Nova-Forge is built on the work of the open-source community that created the engine foundation. Regardless of how far the conversion progresses, the following remain permanent:

- `README.md` credits section acknowledging the inherited foundation and its contributors
- `LICENSE` — GNU GPL v3.0 as inherited; any Nova-Forge-specific additions remain GPL-compatible
- In-game credits screen entry for the engine foundation
- `CHANGELOG.md` historical entries for the rebrand milestone

Removing or obscuring these credits is not permitted at any stage of the conversion.
